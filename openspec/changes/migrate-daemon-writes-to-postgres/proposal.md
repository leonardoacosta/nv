# Proposal: Migrate Daemon Writes to Postgres

## Change ID
`migrate-daemon-writes-to-postgres`

## Summary
Make the Rust daemon (`crates/nv-daemon/`) write obligations, contacts, sessions, and session events directly to Postgres, and consolidate briefing generation into a single Postgres writer. This eliminates the SQLite/JSONL data silo that leaves the dashboard showing empty pages.

## Context
- Extends: `crates/nv-daemon/src/obligation_store.rs`, `crates/nv-daemon/src/contact_store.rs`, `crates/nv-daemon/src/briefing_store.rs`, `crates/nv-daemon/src/orchestrator.rs`, `crates/nv-daemon/src/http.rs`, `crates/nv-daemon/src/worker.rs`
- Related: `openspec/changes/migrate-to-shared-postgres` (Postgres infra migration -- orthogonal, that spec moves the PG instance; this spec makes the daemon write to it)
- Related: `openspec/changes/revive-obligations` (depends on this spec -- obligations must be in Postgres before the dashboard can show them)
- Related: `openspec/changes/rewire-dashboard-api` (dashboard reads from tRPC/Drizzle against Postgres)
- Postgres schemas already exist: `packages/db/src/schema/obligations.ts`, `contacts.ts`, `sessions.ts`, `session-events.ts`, `briefings.ts`
- tRPC routers already exist: `packages/api/src/routers/obligation.ts`, `session.ts` -- they read from Postgres and return empty results because nothing writes there
- The daemon already has `tokio-postgres` and `tokio-postgres-rustls` as dependencies (used in `scheduler.rs` for reading `briefing_hour` from the `settings` table, and in `tools/neon.rs` and `tools/plaid.rs`)
- `DATABASE_URL` env var is the connection string (confirmed in `scheduler.rs`)

## Motivation

The Rust daemon is the sole producer of obligations, contacts, session lifecycle events, and briefings. It writes to three separate local stores:

1. **SQLite** (`~/.nv/messages.db`): obligations and contacts via `rusqlite`
2. **JSONL** (`~/.nv/state/briefing-log.jsonl`): briefing entries via file I/O
3. **In-memory** (`HashMap`): session tracking in `CcSessionManager`, never persisted

Meanwhile, the Next.js dashboard reads from Postgres via tRPC + Drizzle. The Postgres tables (`obligations`, `contacts`, `sessions`, `session_events`, `briefings`) exist with proper schemas but have zero rows -- nothing writes to them from the daemon.

The TS daemon (`packages/daemon/`) does write briefings to Postgres (via `runner.ts`), creating a second briefing writer that diverges from the Rust daemon's JSONL path. This dual-writer situation means briefings generated by the Rust daemon never appear on the dashboard.

This is the foundational fix. Every dashboard page that shows "No data" traces back to this gap.

## Requirements

### Req-1: Introduce a shared Postgres connection pool

Create a `PgPool` abstraction in `crates/nv-daemon/src/pg_pool.rs` that:

- Connects to `DATABASE_URL` at daemon startup using the existing TLS detection pattern from `scheduler.rs` (TLS for Neon/cloud URLs, NoTls for local)
- Maintains a single `tokio_postgres::Client` (or a lightweight pool of 2-3 connections) shared across all Postgres-writing stores via `Arc`
- Spawns the connection driver task (`tokio::spawn(connection)`)
- Provides a `query`/`execute` interface that stores call
- Handles reconnection on connection loss (the daemon is long-running)
- Is injected into `SharedDeps` so all workers and the orchestrator can access it

### Req-2: Obligations -- write to Postgres

Create `PgObligationStore` alongside the existing `ObligationStore`:

- Implement `create()` that INSERTs into the Postgres `obligations` table (columns: id, detected_action, owner, status, priority, project_code, source_channel, source_message, deadline, created_at, updated_at)
- Implement `update_status()`, `update_status_and_owner()`, `update_detected_action()`, `snooze()` that UPDATE the Postgres row
- Implement `get_by_id()`, `list_all()`, `list_by_status()`, `list_by_owner()` for reads
- Implement `update_last_attempt_at()` for the autonomous execution cooldown
- Column mapping: Rust camelCase fields map to Postgres snake_case columns (matching the Drizzle schema in `packages/db/src/schema/obligations.ts`)
- UUID generation uses `uuid::Uuid::new_v4()` (same as current SQLite store)
- Timestamps use `NOW()` (Postgres) instead of `datetime('now')` (SQLite)
- The `owner_reason` field exists in SQLite but not in the Postgres schema -- omit it (the dashboard never reads it)

During the dual-write period, callers in `orchestrator.rs` and `worker.rs` write to both stores. After verification, SQLite writes are removed.

### Req-3: Contacts -- write to Postgres

Create `PgContactStore`:

- Implement `create()` that INSERTs into the Postgres `contacts` table (columns: id, name, channel_ids as JSONB, relationship_type, notes, created_at)
- Implement `update()`, `delete()`, `get()`, `list()`, `search()`, `find_by_channel()`
- Note: Postgres `contacts` schema lacks `updated_at` (SQLite has it). Either add a migration or accept the omission since the dashboard schema is authoritative.
- `channel_ids` is stored as JSONB in Postgres vs JSON text in SQLite -- serialize with `serde_json::to_value()` instead of `serde_json::to_string()`
- `find_by_channel()` uses Postgres JSONB operator (`channel_ids->>'telegram' = $1`) instead of SQLite `json_extract`

### Req-4: Sessions -- write lifecycle events to Postgres

Modify `CcSessionManager` and the orchestrator to write session lifecycle to the Postgres `sessions` table:

- On `CcSessionManager::start()`: INSERT into `sessions` (id, project, command, status='running', trigger_type, started_at=NOW())
- On `CcSessionManager::stop()`: UPDATE `sessions` SET status='stopped', stopped_at=NOW() WHERE id=$1
- On health monitor state transitions (Completed, Failed, Error): UPDATE status and stopped_at accordingly
- The `sessions.id` column is UUID -- use the existing `ta-<uuid>` format or generate a clean UUID (the tRPC router expects UUID format for `getById`)
- Increment `message_count` and `tool_count` as worker events flow through the orchestrator (via `WorkerEvent::ToolCalled` and message completion)

### Req-5: Session events -- write to Postgres

Write session-level events to the `session_events` table (currently has ZERO inserts in the entire codebase):

- On worker dispatch: INSERT event_type='session_start', direction=NULL, content=trigger summary
- On `WorkerEvent::StageStarted`: INSERT event_type='stage_started', content=stage name
- On `WorkerEvent::ToolCalled`: INSERT event_type='tool_call', direction='outbound', content=tool name, metadata={tool_name, duration_ms}
- On `WorkerEvent::Complete`: INSERT event_type='session_end', content=response summary
- On `WorkerEvent::Error`: INSERT event_type='error', content=error message
- Use the `session_id` FK pointing to the parent `sessions` row
- Batch inserts where possible (e.g., buffer events and flush every 5 seconds or on session end) to avoid per-event connection overhead

### Req-6: Briefings -- consolidate to single Postgres writer

Eliminate the JSONL briefing path in the Rust daemon:

- In `orchestrator.rs` where `briefing_store.append(&entry)` is called (line ~2355), replace with a Postgres INSERT into the `briefings` table (columns: id, generated_at, content, sources_status as JSONB, suggested_actions as JSONB)
- Match the TS daemon's INSERT pattern from `packages/daemon/src/features/briefing/runner.ts` (which already writes to Postgres)
- Remove the `BriefingStore` (JSONL) dependency from `SharedDeps` and `HttpState`
- Remove the `/api/briefings` endpoint in `http.rs` that reads from the JSONL store (the dashboard now reads via tRPC from Postgres)
- Keep the `BriefingEntry` type as-is (it maps cleanly to the Postgres columns)

### Req-7: Dual-write migration strategy

Implement a phased migration to avoid data loss:

**Phase 1 -- Dual-write (this spec):**
- All stores write to both SQLite/JSONL AND Postgres
- Reads continue from SQLite (daemon internal) and Postgres (dashboard)
- Add a `pg_write_enabled` flag in daemon config (default: true) so Postgres writes can be disabled without code changes if issues arise
- Log discrepancies at warn level if a Postgres write fails (SQLite write is the fallback)

**Phase 2 -- Verify (manual, post-deploy):**
- Compare row counts between SQLite and Postgres for obligations and contacts
- Verify dashboard pages show data
- Monitor Postgres write latency in daemon logs

**Phase 3 -- Cut over (follow-up spec):**
- Remove SQLite writes for obligations and contacts
- Remove JSONL writes for briefings
- Remove `BriefingStore`, simplify `ObligationStore` and `ContactStore` to Postgres-only
- Remove `rusqlite` dependency from `obligation_store.rs` and `contact_store.rs` (keep it for `MessageStore` and `conversation_db` which remain SQLite)

### Req-8: Error handling and resilience

- Postgres writes MUST NOT crash the daemon. All PG operations are wrapped in `Result` and failures are logged at `warn` level.
- If the Postgres connection drops, the daemon continues operating with SQLite as the source of truth. Reconnection attempts happen on the next write.
- For session events, use fire-and-forget async writes (spawn onto the tokio runtime) to avoid blocking the orchestrator loop.
- Connection timeout: 5 seconds (matching the existing pattern in `scheduler.rs`).

## Scope
- **IN**: PgPool abstraction, PgObligationStore, PgContactStore, session writes, session_event writes, briefing Postgres writes, dual-write wiring in orchestrator/worker, config flag, error handling
- **OUT**: Removing SQLite stores (Phase 3 follow-up), TS daemon changes (already writes to PG), dashboard changes (already reads from PG), schema migrations (PG tables already exist), `MessageStore` migration (stays SQLite -- high-volume, local-only), `conversation_db` migration (stays SQLite)

## Risks
| Risk | Mitigation |
|------|-----------|
| Postgres unavailable at daemon start | Daemon starts anyway; PG writes disabled until connection succeeds. SQLite remains functional. |
| Schema mismatch between Rust types and Drizzle schema | Column names verified against `packages/db/src/schema/*.ts` in this proposal. Types are primitive (text, integer, timestamp, jsonb). |
| Dual-write consistency | SQLite is the primary during Phase 1. PG failures are logged but do not block. Phase 2 reconciliation catches drift. |
| Connection pool exhaustion | Single client (or 2-3 pool) is sufficient for daemon write volume (~100 writes/day for obligations, ~1/day for briefings, ~10/day for sessions). |
| UUID format mismatch | SQLite stores use `Uuid::new_v4().to_string()` (hyphenated). Postgres UUID columns accept this format natively. |
| `owner_reason` field loss | Field exists only in SQLite, not used by dashboard. Acceptable to omit from PG writes. Document in follow-up cleanup spec. |
| Session ID format | `CcSessionManager` uses `ta-<uuid>` format. Postgres `sessions.id` is UUID type. Use the inner UUID portion, or switch to clean UUIDs for PG writes (map back to `ta-` prefix for internal use). |

## Alternatives Considered
1. **Sync daemon (SQLite to Postgres)**: Rejected -- adds a third component, sync lag, conflict resolution complexity. Direct writes are simpler.
2. **Replace SQLite entirely in one shot**: Rejected -- too risky for a single spec. Dual-write gives a safety net.
3. **Have the dashboard read from the daemon API instead of Postgres**: Rejected -- the dashboard is already built on tRPC/Drizzle/Postgres. Changing the read path is more work than fixing the write path.
4. **Use sqlx instead of tokio-postgres**: The daemon already uses tokio-postgres. Adding sqlx introduces a second PG driver. Not worth the churn.
